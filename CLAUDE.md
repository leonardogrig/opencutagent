# Working in OpenCutAgent (read this first)

OpenCutAgent (formerly EditAgent; the local folder is still `editagent/`) is a Premiere Pro CEP panel + Node MCP server that lets Claude edit a video on Premiere's **live timeline** (transcribe → mark retakes/silences → cut). The "AI analysis" (retake keep/cut, silence threshold) is done by **Claude itself** — either you in this chat via the `ppro_*` MCP tools, or, when the panel runs in its default mode, by the server spawning a headless `claude -p` call on the user's subscription. There is no external/paid LLM.

**Golden rule:** before driving the `premiere` MCP, skim **Lessons learned** below. Most failures here are known and have a one-line fix. When you hit a *new* one, **append it** (see "Record a new lesson"). The point is to never burn a session re-discovering a dead end.

## Architecture (so symptoms make sense)
`Claude Code ──stdio──► MCP server (server/, hosts ws on 127.0.0.1:3001 + runs ffmpeg/Scribe) ──ws──► CEP panel (cep-panel/) ──evalScript──► cep-panel/host/premiere.jsx` (all host ops live here). Server state `ctx.review` holds the loaded segments + keep/cut marks; `ctx.silence` holds the loudness session.

**Panel AI (two modes, set by the "Sync with Claude Code" toggle in the panel's AI bar):** OFF (default) → the panel buttons call the `aiThreshold` / `aiRetakes` RPCs, which run `claude -p` **headlessly** from the server (`server/ai.js`) and apply the answer — the user never types in chat. ON → the buttons defer to *this* chat: you call the `ppro_*` tools and push `silenceConfig`/`reviewUpdate` to the panel (the original flow). Both share `ctx.silence`/`ctx.review`, so results render identically. Model/effort come from the panel dropdowns (persisted in localStorage), overridable via `.env` (`EDITAGENT_AI_MODEL`/`_EFFORT`).

**Who starts the server (the #1 source of "why isn't it working"):** the panel is ONLY a WS client; nothing works until `server/index.js` is listening on 3001. It can be started three ways, and they must not collide: (1) **the extension auto-starts it** — `main.js tryAutostartServer()` spawns `node server/index.js` when a connect attempt fails (resolves the symlinked extension path with `fs.realpathSync`, finds node in common paths), so just opening the panel is enough for **headless** mode; (2) **Claude Code's MCP** spawns it via `.mcp.json` (stdio) — needed for **Sync** mode (chat ↔ MCP); (3) **manually**: `npm start` (root), `node server/index.js`, or `./start-opencutagent.command` (stop with `npm stop`). It's ONE server per machine on port 3001 operating on whatever sequence is active — NOT per video project. Headless needs the server but NOT a chat. For Sync, start Claude Code FIRST (so its server owns 3001), then open the panel (it connects instead of auto-starting). Auto-start is check-first, so it won't duplicate a server that's already up.

## Normal retake workflow
1. Confirm connected (a `ppro_get_timeline_state` that returns data). 
2. `ppro_get_retake_segments` → populates server `ctx.review`; analyze the segments yourself (keep the last complete take of each beat; cut fragments/false-starts/filler).
3. `ppro_mark_retakes` (one batched call; generate the cut list with a script for hundreds) → marks push live to the panel.
4. `ppro_apply_retakes` (`remove_gaps:true` = ripple/tight) **or** the panel's Apply All.
5. **VERIFY by re-reading the timeline** (clip count + duration), not the "applied" count.

## Lessons learned — READ before using the MCP (append when you learn a new one)

- **"panel not connected" though the panel shows Connected / PID churn / "No segments loaded":** stale server instances are squatting port 3001 while this session's server can't bind (no panel), and/or the MCP client is respawn-thrashing (common after a time gap). Fix: `pkill -9 -f "editagent/server/index.js"` **once**, then reset the *client* with `/mcp` → reconnect `premiere` (or restart Claude Code). Verify exactly one listener: `lsof -nP -iTCP:3001 -sTCP:LISTEN`. Do NOT repeatedly kill+probe — every kill triggers a respawn and makes it worse. The server *code* is fine; this is connection state.
- **"Unknown RPC method: <name>" after a code change (e.g. clicking a new panel button):** the panel (new client code) is talking to a server running OLD code. Root cause: **more than one Claude Code window open in the project** — each spawns its own `server/index.js`; the *oldest* one that grabbed port 3001 keeps serving the panel with its stale in-memory code, and a restarted window's new server can't bind (gives up after ~6s of retries) so it's never reached. Check: `ps -eo pid,lstart,command | grep "[c]laude$"` (count windows) and `lsof -nP -iTCP:3001 -sTCP:LISTEN` (who holds it) + that PID's start time. Fix: keep **ONE** Claude Code window in the project — close the others, then in the survivor `/mcp` → reconnect `premiere` (or restart it) so a fresh new-code server binds 3001. Killing the squatter frees the port but nothing auto-rebinds (a server that already gave up won't retry; the client must respawn it).
- **Loading code changes:** server (`server/*`) → **restart Claude Code** (no hot-reload, loses `ctx.review`). Host script (`cep-panel/host/premiere.jsx`) → reopen the panel, OR hot-patch live with `$.evalFile("<absolute repo path>/cep-panel/host/premiere.jsx")` via `ppro_run_script` (no reopen, keeps `ctx.review`). Panel UI (`cep-panel/client/*`) → reopen the panel.
- **Panel design system (2026-06-28):** the panel UI was redesigned around a token system in **`cep-panel/client/styles.css`** (linked stylesheet = single source of truth), specced in **`cep-panel/DESIGN.md`**, QA'd via **`cep-panel/client/preview.html`** (component gallery: `python3 -m http.server --directory cep-panel/client`, open `preview.html` to see every component/state). ONE accent = **Ember `#EC6A41`** (warm coral, intentionally NOT Anthropic's `#D97757`), replacing the old blue-vs-orange two-primary clash; bundled **Inter** (`client/fonts/`). main.js UX helpers: loading-spinner buttons (`.loading` via `setLoading`/`setLabel` — dynamic buttons keep `<svg class="ic">`+`<span class="lbl">` so a label swap never wipes the icon), `#topbar` busy bar (`setBusyBar`), typed `toast(msg,'success'|'error'|'info')`, `skeletonRows()` on transcribe. Canvas colors are hex literals in `draw()`/`drawOverview()` (canvas can't read CSS vars) — keep synced to DESIGN.md. **CSP** needs `font-src 'self'` + `img-src 'self' data:`. **Client-only** (server/host untouched; `npm test` passes). Verify CSS/markup edits by reopening the panel (or the preview.html gallery in a browser).
- **`.mcp.json`:** this Claude Code build does NOT expand `${CLAUDE_PROJECT_DIR}`/`${VAR}`. Use an absolute path for the server, no `env` block.
- **Drop-frame detection:** only **true** 29.97/59.94 are drop-frame. A "30 fps" sequence is often **30.00003** (non-integer timebase) and is **non-drop**. A loose drop-frame check builds drop timecodes that razor a few frames off → "razor everywhere, delete nothing." (Fixed in `detectDropFrame`.)
- **Apply reporting lies:** "applied X/X" counts host *calls*, not actual deletions. Always verify against the real timeline (clip count/duration) before declaring success.
- **Failed apply = sliced timeline:** a bad apply razors the clip into hundreds of pieces with nothing removed. Don't try to undo hundreds of steps. Recover by making a fresh sequence from the source clip (untouched in the Project bin: right-click → New Sequence From Clip).
- **ElevenLabs key** must have the **speech_to_text** permission scope, or transcription 401s.
- **`ppro_get_retake_segments` on a long recording** can exceed the MCP output cap; the full result is auto-saved to a file (read it with `jq`), or rely on the compact output. `ctx.review` is still populated even when the return overflows.
- **`.env` changes** (API key / model) take effect on the next call without a restart (`liveEnv`); code changes do not.
- **Cut decisions are persisted** at `.cache/retake-decisions.json` (330 cuts, by index) for the in-progress `2026-06-17_17-19-37` edit — re-mark from there instead of re-analyzing.
- **Panel stuck on "Waiting for server…" / "Waiting for Claude Code":** the server isn't running on 3001 (the panel is just a client). Not a code bug. Fastest check: `lsof -nP -iTCP:3001 -sTCP:LISTEN`. Fix: reopen the panel (it now auto-starts the server), or `node server/index.js` / `./start-opencutagent.command`, or `/mcp` reconnect `premiere` in a Claude Code session. See "Who starts the server" above. (Headless AI does NOT remove the need for the server — only the need to type in chat.)
- **Headless panel AI (`server/ai.js`):** the server's "do it for me" buttons spawn `claude -p` as a pure judgment oracle (data in → JSON out; it does NOT touch Premiere). Non-negotiable flags so it can't fight the running server: `--strict-mcp-config` (loads NO `.mcp.json`, so it never spawns a 2nd `premiere` bridge on 3001) + `--tools ""` (no file/bash) + cwd=`tmpdir()` (skips project CLAUDE.md). Subscription auth: do **NOT** use `--bare` (it never reads keychain OAuth → forces an API key); plain `-p` reads the login. We `delete env.ANTHROPIC_API_KEY` so a stray key can't silently bill. Structured output via `--json-schema` → read `.structured_output`. There is **no `--max-turns`** in this CLI build (don't add it). Prompt goes on **stdin** (avoids ARG_MAX on long transcripts); skills are injected with `--system-prompt` reading the actual `.claude/skills/*/SKILL.md` (single source of truth).
- **Retakes "0 to cut" / "isn't efficient" / weird sub-sentence fragments:** "0 to cut" = **no analysis ran** — Load/Reload only transcribes; cuts come from "Analyze w/ Claude" (headless) or Sync chat. Segmentation is **clip-bounded** (`review.js buildReview`): transcript parsed once per source (`wordsByMedia`), sliced to each clip window (`sliceWordsToWindow`, midpoint rule), phrased *within* the clip, then `partitionClip` tiles `[sourceIn,sourceOut]` with no gaps — a segment never crosses a cut, cutting leaves no slivers, and a word-empty clip still yields one cuttable "(no speech)" segment. Word-empty clips (pops/breaths) → `fragment:"empty"` → **auto-marked Cut**; sub-0.5s spoken clips → `fragment:"short"` → **flagged only** (kept, reviewable). No-speech cuts are **deterministic, not the AI's call**: headless "Analyze w/ Claude" (`aiRetakes`) never sends empty segments to `claude -p` and skips them in its mark-reset, so the AI can't un-cut them — it only judges real speech (the Sync/`ppro_mark_retakes` path doesn't reset, so empties stay cut there too). Transcription is correctly FILE-based + cached (matches Premiere TBE/Descript/TimeBolt — transcribe SOURCE once, MAP onto timeline); "Reload" reuses the cache by design — pass `refresh:true` only if the source file truly changed. Do NOT "fix" by transcribing per-clip (worse accuracy + N API calls; slice the one full transcript instead). Tests: `server/test/retakeSegments.js`.

- **Headless "Analyze w/ Claude" keeps duplicate retakes / hangs on a long timeline (2026-06-29):** symptom → the panel kept obvious **serial restarts** (the speaker re-starting the same line, getting further each time — "n8n now has-" → "…a set of skills-" → … all kept) and on a full 400+-segment timeline the single `claude -p` call **timed out (>900s)** with no answer. Cause → (1) one giant call must emit hundreds of cut decisions in one structured response → it gets lazy mid-list or times out; (2) the retake guidance under-specified the #1 pattern and "BIAS TOWARD KEEP when unsure" made it keep duplicates. Fix → `aiRetakes` now calls **`analyzeRetakes` (server/ai.js)**, which **chunks** the timeline into overlapping windows (each owns a ~36-segment block + sees a ~14-segment context margin each side so a boundary-straddling restart-run is judged with the line's completion visible; concurrency-capped; owner-disjoint merge = no conflicts), turning one unreliable call into ~12 small reliable ones. The skill's "How to judge retakes" section + `retakeSystem()` judgment were rewritten to name the serial-restart pattern, keep only the most-complete pass of each run, distinguish restarts (same words) from distinct next-points (new words → **multiple keepers per beat allowed**, NOT "exactly one"), and reserve keep-when-unsure for genuine content ambiguity. Tune with `EDITAGENT_AI_CHUNK` / `_CHUNK_CONTEXT` / `_CONCURRENCY` (.env). **Eval:** `npm run eval:retakes` (`server/test/eval/`, real `claude -p` over the real chunked path, scored vs a human-labeled 412-segment fixture) → went from **timeout → 97% F1 (beat-level: 3 leftover dupes / 4 lost of 412)**. `planRetakeChunks` unit-tested in `server/test/retakeSegments.js`. NOTE: verified via the eval, not yet in live Premiere.

- **Apply All took ~30 min on a 2h video (2026-07-02, full-stack, NOT yet live-verified):** symptom → Remove Silences "Apply All" (and retakes Apply All) crawled: one `removeRange` host call **per range** = an evalScript round-trip + QE razor pass + a **RIPPLE delete that shifts every downstream clip** → O(ranges × clips) DOM work. Fix → batched **razor → lift → close**: new host ops **`removeRangesBatch`** (razor ALL edges first — razors never shift, so every timecode stays valid; then lift-delete (`clip.remove(false,false)`) every piece inside a range, descending index, binary-search range match) and **`closeRangeGaps`** (per track, a range counts toward the shift only when actually EMPTY there — self-verifying, failed razors can't mis-shift; each surviving clip `move()`s left exactly ONCE by the emptied duration before it, ascending order; pre-existing gaps untouched). Shared server helper **`applyRangesBatched`** (`silences.js`): merges ranges (`mergeFrameRanges`), chunks ~50/host-call for progress + cancel (nothing shifts during lift, so chunk order is free; on abort it still closes the processed chunks' gaps), timeouts 180s/chunk + 600s/close. Wired into `applySilenceRanges` (remove/keepSpaces; mute stays per-range), `applyReview` (retakes), and `ppro_remove_silences`. Bonus correctness: a pre-existing edit point inside a silence now removes ALL pieces (old code only removed the piece at startFrame). Tests: `silenceApply.js` (batch/chunk/merge), `feature.js`, `cancelUndo.js` + a scratch sim of the shift math. `removeRange` kept for compat. Host change → reopen panel (or `$.evalFile`); server → restart Claude Code.

- **"Removed 0/N silence range(s) … ~0s" and nothing changed (2026-07-02):** symptom → fast Apply All reports 0 applied instantly, timeline untouched. Cause → **stale host script**: Premiere's ExtendScript engine still had the old `premiere.jsx` (reopening the panel does not reliably re-eval ScriptPath), so every `removeRangesBatch` call failed "Unknown action" — and those per-chunk errors were only in `result.errors`, which the panel never displays. Fixes (server): `callBatchOp` (silences.js) now **self-heals** — on "Unknown action" it hot-patches the live engine via the runScript op (`$.evalFile` of the repo jsx) and retries once; apply messages now append `N error(s) — first: …` so failures are never a silent "0/N". - **XML-rebuild fast apply — TimeBolt-style (2026-07-02, full-stack, NOT yet live-verified):** in-place razoring has a hard floor (QE razor is per-track only, no multi-track razor, every DOM op pays Premiere bookkeeping + redraw), so big Apply Alls now **skip the timeline entirely**: `server/rebuild.js` builds the FINISHED tightened sequence as **FCP7 XML (xmeml v4)** — pure `computeKeepLayout` (clips minus cuts, global compaction shift → A/V sync by construction, pre-existing gaps preserved, BigInt-exact int frames) + `buildXmeml` (integer timebase + ntsc flag TRUE only for true /1.001 rates; one full `<file>` def then id refs; audio `<sourcetrack>` trackindex = original track+1; V/A spans from the same source range are `<link>`ed; XML-escaped names + percent-encoded `pathurl`) — then host op **`importXmlSequence`** (`app.project.importFiles([xml], true, rootItem, false)` + find newest sequence by name + `openSequence`). Wired as the DEFAULT in `applyRangesBatched` when ripple && cuts ≥ **`EDITAGENT_REBUILD_MIN`** (default 100; 0 disables): 2443 cuts → 4888-clipitem 2.7MB XML in **13ms** + one import (~seconds) vs minutes of razoring. **Any rebuild failure falls back silently to the razor path** — including v1's deliberate refusals: any timeline item without source media (titles/graphics/adjustment layers) or non-100%-speed clip throws in computeKeepLayout. Result is a NEW sequence "`<name> - tightened`" (original untouched → `undoable:false`, no captureUndo; "undo" = delete it); clip effects/grades do NOT travel through xmeml. `getTimelineState` now also returns `frameSize`. Self-heal helper moved to `tools/util.js callHostHealing` (shared by batch + import ops). Tests: `test/xmlRebuild.js` (27 checks: layout/emission/routing/fallbacks). LIVE-VERIFY on first run: import relinks media (not offline), audio channels sane, frame-accurate at the 30.00003 timebase, links behave.

- **Round-trip XML rebuild — fast apply now PRESERVES effects (2026-07-03, full-stack, LIVE-VERIFIED same day — effects/transforms survived a real apply):** symptom → after a big Apply All the user's clip configs (Motion position/scale etc.) were gone: the v1 rebuild GENERATED a bare xmeml from our own bookkeeping, so nothing but clip spans survived. Fix → the fast path now **round-trips Premiere's own XML**: new host op **`exportXmlSequence`** (`seq.exportAsFinalCutProXML(path)`, try-ladder on a 2nd suppressUI arg) → **`server/roundtrip.js`** edits TIMING ONLY in that document and reimports. Core = a minimal identity-preserving XML tree (`parseXml`/`serializeXml` — raw attr strings, comments/CDATA verbatim; `serialize(parse(x))===x`) + pure **`transformXmeml(xml, cuts, {sequenceName})`**: per track, each clipitem is split/trimmed/deleted via the shared `keepSpans`/`makeRemovedBefore` (now exported from rebuild.js) and compacted; every node we don't understand (Motion/opacity/volume `<filter>`s, labels, third-party metadata) passes through VERBATIM. Gotchas handled: **`pproTicksIn/Out` recomputed** from the new in/out (stale ticks would beat the frame values on import; non-integral ticks-per-frame → elements stripped), **split-clip `<link>`s** re-suffixed deterministically (`-p2`, `-p3` — V/A partners split identically so refs stay paired), **full `<file>` defs rescued** into the first surviving reference when the def-holding clip is deleted, **`<uuid>` stripped** (collides with the source sequence) + `<name>` → "… - tightened", transitions dropped when a cut overlaps them / shifted otherwise, sequence markers shifted or dropped, sequence `<duration>` updated. v1 refusals (throw → fallback): clipitem `start`/`end` = **-1** (transition-relative timing), missing start/end/in. Tests: `test/xmlRoundtrip.js` (33 checks: parser identity, timing surgery, ticks, links, file rescue, transitions, markers, routing); `xmlRebuild.js`'s fake ctx now rejects `exportXmlSequence` so it keeps testing the generated path. **"Translation Report" alert on apply → benign, from the EXPORT step:** Premiere logs intrinsic untranslatables (e.g. "MPEG Source Settings", an MPEG source-interpretation entry it drops from the XML and re-derives from the media on import) to "FCP Translation Results….txt" next to the XML and pops a modal.
